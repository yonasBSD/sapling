# SCS Reduced Availability Workarounds

This document explains why SCS reduced availability should not cause SEV1-level incidents, and what alternatives exist for production-critical operations that do not require source control.

## Background

SCS (`mononoke-scs-server`) and Mononoke Server / SLAPI are both **criticality-2** services. Mononoke as a whole is developer-facing infrastructure: it serves source control operations for engineers writing, reviewing, and landing code. It is not in the production serving path.

Because Mononoke is developer-facing, its reduced availability **cannot cause a SEV1**. A SEV1 requires production user impact. Source control outages affect developer productivity and code velocity, but production systems do not depend on source control to serve traffic or maintain a healthy state.

## Why Source Control Is Not Needed for Production Recovery

During a production incident, the goal is to restore production to a known good state. This does not require source control:

- **Conveyor rollback**: Production deployments can be rolled back to a previous known-good version through Conveyor without any source control interaction. Rollbacks are the primary mechanism for production recovery.
- **Deployment systems operate independently**: Once code is built and deployed, the deployment pipeline does not depend on Mononoke to maintain or revert the running state.
- **Configuration changes**: Emergency config changes go through Configerator, not source control.

Landing a new fix through source control is one way to resolve a production issue, but it is never the **only** way. If landing a fix is blocked by a source control outage, a rollback achieves the same goal (restoring production health) without source control.

## What SCS Outages Actually Affect

SCS outages affect **developer-facing workflows**:

- Phabricator web UI displaying commit contents and diffs
- CodeHub source browsing
- SCMQuery-based tooling and automation
- Batch jobs (codesearch indexing, Myles)

These are productivity degradations, not production incidents.

## For Service Owners: Removing Incorrect Criticality Dependencies

If your service or workflow treats a source control outage as a SEV1 trigger, the dependency model is incorrect. Source control is developer infrastructure, not production infrastructure.

Review your incident response procedures:
1. **Ensure rollback capability**: Every production service must be able to roll back to a previous version without source control. If your only recovery path is "land a fix," you have a gap in your deployment tooling, not a source control problem.
2. **Do not gate production recovery on code review**: During an active incident, if a rollback is available, use it. Do not wait for a code fix to go through review and land.
3. **Separate deployment from source control**: Deployment systems should depend on build artifacts and deployment infrastructure, not on live source control queries.
